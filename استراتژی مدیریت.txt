استراتژی مدیریت State (State Management Strategy)
ما دو نوع دیتا داریم. قانون این است که هر کدام را با ابزار خودش مدیریت کنیم:

Server State (داده‌های روی سرور Supabase):
این شامل لیست پروژه‌ها، مشتریان، درخواست‌ها و رزومه‌ها می‌شود. مسئولیت Fetch، Cache و بروزرسانی این داده‌ها کاملاً با React Query است. ما فقط به React Query می‌گوییم چه چیزی را از کجا بخواند، و بقیهٔ کارها (لودینگ، خطا، کش) را او برایمان انجام می‌دهد.

Client State (داده‌های مربوط به خود اپ):
این شامل وضعیت لاگین کاربر، نقش کاربر (Admin/Client) و تم (تاریک/روشن) می‌شود. برای مدیریت این وضعیت‌های سراسری و پیچیده، از معماری "گوینده و بلندگو" (Context API + useReducer) خودت استفاده می‌کنیم.

۵. جریان کلی داده (Data Flow Overview)
وقتی یک صفحه (مثلاً داشبورد پروژه‌ها) بارگذاری می‌شود، چه اتفاقی می‌افتد؟

کاربر (Admin) به /app/projects می‌رود.

ProtectedRoute بررسی می‌کند که کاربر لاگین کرده و نقش Admin دارد.

کامپوننت ProjectsBoard با استفاده از React Query یک درخواست به Supabase می‌فرستد: «همهٔ پروژه‌ها را بده».

در حین دریافت داده، React Query یک isLoading = true برمی‌گرداند و ما یک <Spinner /> نشان می‌دهیم.

پس از دریافت داده، آن را در جدول نمایش می‌دهیم.

اگر خطایی رخ دهد، یک پیام خطا نمایش می‌دهیم.

این نقشه، تمام آن چیزی است که برای شروع Micro-Task اول نیاز داریم. هر تغییری که در طول مسیر لازم شود، به همین سند اضافه یا اصلاح خواهد شد.
